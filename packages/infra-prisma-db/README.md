# @interviews-tool/infra-prisma-db

Prisma as a **parallel ORM layer** over the exact same Neon tables that
`@interviews-tool/infra-db` (Drizzle) already uses. This package exists to demonstrate that
the same data can be read/written with **Drizzle or Prisma**, interchangeably, without
changing any table or breaking production.

Prisma owns nothing here: the schema is produced by **introspection** (`prisma db pull`) and
no Prisma migration ever runs against the existing tables.

## Requirements

- The same `DATABASE_URL` / `DATABASE_URL_DIRECT` (Neon) used by the rest of the monorepo,
  synced via dotenvx from `apps/server/.env`.
- Prisma **7** (`prisma`, `@prisma/client`, `@prisma/adapter-neon`), already declared here.

## Prisma 7 layout (important)

Prisma 7 removed the connection `url` from `schema.prisma`. Instead:

- `prisma/schema.prisma` — declares only the `provider` and the generated models.
- `prisma.config.ts` — holds the CLI connection URL (`datasource.url`, the **direct**
  non-pooler Neon URL) for `db pull` / `studio`.
- `src/client/index.ts` — the runtime client connects via the **Neon HTTP driver adapter**
  (`@prisma/adapter-neon`), the same fetch-based transport Drizzle's `neon-http` client uses.
  No `url` in the schema; the adapter carries the connection.

### Two generated clients (workerd + node)

The apps (`apps/server`, `apps/web`) run on **Cloudflare Workers (workerd)**, while the local
example scripts run on **Bun**. A single client build can't serve both: the workerd client
uses a WASM query compiler that Bun can't instantiate, and the node client uses
`fileURLToPath(import.meta.url)` which crashes on workerd. So `schema.prisma` declares two
generators:

- `client` → `src/generated` (`runtime = "workerd"`) — imported by `src/client/index.ts`
  (`createPrismaClient`). This is what the apps and the `PrismaHiringProcessRepository` use.
- `clientNode` → `src/generated-node` (`runtime = "nodejs"`) — imported by
  `src/client/node.ts` (`createNodePrismaClient`). Used **only** by the example scripts.

Both are committed (pure TypeScript, no engine binaries) and regenerated by
`bun run prisma:generate`.

## Workflow

All commands inject the DB URL via dotenvx (or inline). From the repo root:

```bash
# 1. Introspect the existing Neon tables into prisma/schema.prisma
bun run prisma:pull

# 2. Generate the typed Prisma clients into src/generated + src/generated-node
bun run prisma:generate

# 3. Run the side-by-side Drizzle-vs-Prisma demo (raw clients)
bun run prisma:example

# 4. Run the repository-swap demo (same use case, either repository)
bun run prisma:example:repo
```

Or from this package directory, injecting the env yourself:

```bash
DATABASE_URL_DIRECT="postgresql://...ep-...neon.tech/neondb?sslmode=require" \
  bunx prisma db pull            # writes models into prisma/schema.prisma

bunx prisma generate             # no DB needed

DATABASE_URL="postgresql://...-pooler...neon.tech/neondb?sslmode=require" \
  bun run examples/compare-orms.ts
```

> The generated clients (`src/generated`, `src/generated-node`) are committed, so a fresh
> clone builds without a codegen step. After changing the schema, run `bun run prisma:generate`
> and commit the regenerated output.

## Usage

```ts
import { createPrismaClient, db } from "@interviews-tool/infra-prisma-db/client";

// Default client bound to process.env.DATABASE_URL:
const rows = await db.interviews_tool_hiring_process.findMany({ take: 10 });

// Or build one with an explicit URL (e.g. per-request on the edge):
const prisma = createPrismaClient(env.DATABASE_URL);
```

Model names come straight from introspection, so they match the real (prefixed, snake_case)
table names — e.g. `interviews_tool_hiring_process`, with columns like `company_name`. You
can tidy these later with `@@map` / `@map` on each model (stable across future `db pull`s);
left as-is here to keep introspection output faithful.

### Repository swap (domain port + two adapters)

The domain defines the port `IHiringProcessRepository`
(`@interviews-tool/domain/repositories`). Two infrastructure adapters implement it over the
**same** table:

- `HiringProcessRepository` — Drizzle (`@interviews-tool/infra-db/repositories`)
- `PrismaHiringProcessRepository` — Prisma (`@interviews-tool/infra-prisma-db/repositories`)

Because both satisfy the same contract, an application use case is served by either one
without change — only the injected adapter differs:

```ts
import { listHiringProcesses } from "@interviews-tool/application/hiring";
import { PrismaHiringProcessRepository } from "@interviews-tool/infra-prisma-db/repositories";
import { createPrismaClient } from "@interviews-tool/infra-prisma-db/client";

const repo = new PrismaHiringProcessRepository(createPrismaClient(env.DATABASE_URL));
const result = await listHiringProcesses({ repo, userId, pagination: { page: 1, limit: 5 } });
```

`bun run prisma:example:repo` runs `listHiringProcesses` through both repositories and prints
identical output (same rows, same domain `status` values). See
`examples/repository-swap.ts`.

### Enum representation note

Drizzle's domain layer maps some enum values (e.g. the DB value `first_contact` is surfaced
as `first-contact` in the app), while Prisma returns the raw database enum value
(`first_contact`). Same underlying row — different presentation per layer. The demo prints
both so the difference is visible.

## Migrations (documented, not executed)

To honor "change nothing," this package runs **no** migrations against the DB. If you later
want Prisma to author schema changes without disturbing the Drizzle-owned tables, use the
standard **baseline** flow (nothing runs against production):

```bash
# 1. Ensure schema.prisma reflects the current DB
bun run prisma:pull

# 2. Capture the current state as an initial migration (SQL only, not applied)
mkdir -p prisma/migrations/0_init
bunx prisma migrate diff \
  --from-empty \
  --to-schema-datamodel prisma/schema.prisma \
  --script > prisma/migrations/0_init/migration.sql

# 3. Mark it as already applied so Prisma never re-creates existing tables
bunx prisma migrate resolve --applied 0_init
```

From there, future `prisma migrate dev --create-only` changes are authored as new migrations
you review before applying.

## Server route (per-endpoint opt-in, live on workerd)

In `apps/server/src/routes/hiring-processes.ts` the Drizzle wiring is untouched; a Prisma
repository is added alongside it, and **individual endpoints opt in**:

```ts
export const hiringProcessRoutes = new Elysia({ prefix: "/hiring-processes" })
  .use(errorHandlerPlugin)
  .decorate("db", createDatabaseClient(env.DATABASE_URL))
  .decorate(
    "hiringProcessRepoPrisma",
    new PrismaHiringProcessRepository(createPrismaClient(env.DATABASE_URL)),
  )
  .derive(({ db }) => ({ hiringProcessRepo: new HiringProcessRepository(db) }))
  // ...
```

Only the list endpoint uses Prisma; every other endpoint keeps using `hiringProcessRepo`
(Drizzle):

- `GET /api/v1/hiring-processes` → **Prisma** (`hiringProcessRepoPrisma`)
- `GET /api/v1/hiring-processes/:id` → Drizzle
- `POST/PUT/DELETE` → Drizzle

To move another endpoint to Prisma, swap `hiringProcessRepo` → `hiringProcessRepoPrisma` in
just that handler. Both implement the same domain port over the same tables, so the use cases
are unchanged.

Verified under the real Workers runtime (`wrangler dev`): the Worker boots with the Prisma
client instantiated on workerd, `/api/v1/health` returns 200, and the endpoints return `401`
without a session (auth-guarded). To see rows, call the list endpoint with a logged-in
session.

To run the server locally against Neon, provide the env to wrangler (it reads
`apps/server/.dev.vars`), e.g. `cp apps/server/.env apps/server/.dev.vars`, then
`cd apps/server && bunx wrangler dev`.

### Deploy note

The generated clients are committed (pure TypeScript, no engine binaries), so app builds and
`wrangler deploy` have everything they need without a codegen step. If you change the schema,
re-run `bun run prisma:generate` and commit `src/generated` + `src/generated-node`.
