# Design: `@interviews-tool/infra-prisma-db` — Prisma as a parallel ORM layer

**Date:** 2026-07-06
**Status:** Approved (design)

## Purpose

Add a new infrastructure package, `@interviews-tool/infra-prisma-db`, that exposes a
**Prisma-based typed client over the exact same tables** the current Drizzle package
(`@interviews-tool/infra-db`) already uses on the same Neon Postgres database.

The goal is a demonstration: a consumer can read/write the same data using **Drizzle OR
Prisma**, interchangeably, without breaking anything in production. Prisma owns nothing —
it is a parallel abstraction layer generated by introspecting the existing schema.

## Hard constraints

- **Do not change any existing table.** No Prisma-authored migrations run against the DB.
- **Same database, same `public` schema, same tables** (`interviews_tool_*` + auth tables).
- **Do not modify `infra-db`** (Drizzle) or any existing consumer.
- All new code is **additive**.
- Reuse the same `DATABASE_URL` (Neon), synced via dotenvx from `apps/server/.env`.

## Context (current state)

- ORM today: Drizzle (`@interviews-tool/infra-db`) + Neon serverless HTTP driver.
- Tables are prefixed `interviews_tool_` (via `pgTableCreator`), plus Better-Auth tables.
- The Drizzle client is consumed in:
  - `apps/server` (Elysia on Bun) — Node-compatible runtime.
  - `apps/web` server functions — **deploys to Cloudflare Workers (edge/workerd)**.
- `infra-db` exports `./client`, `./repositories`, `./schemas`.

## Runtime decision (why the modern Prisma client)

Because the DB client is used at the **edge** (Cloudflare Workers) in `apps/web`, to reach
parity with Drizzle (usable in backend **and** frontend/serverFn) Prisma must avoid the
classic binary query engine, which does not run on Workers.

Approach:

- Use Prisma's modern **`prisma-client` generator** (ESM, no binary engine) instead of the
  legacy `prisma-client-js`.
- Use the **`@prisma/adapter-neon` driver adapter** over the same `DATABASE_URL`.
- Enable the `driverAdapters` capability (GA in Prisma 6.7+; enable as a preview feature if
  the installed version predates GA). Exact flags pinned at implementation time against the
  installed Prisma version.

The same adapter-based client runs on Bun (server) and workerd (edge).

## Package structure (mirrors `infra-db` layer-first style)

```
packages/infra-prisma-db/
├── package.json
├── tsconfig.json
├── .gitignore
├── prisma/
│   └── schema.prisma        # generated by `prisma db pull`, tidied with @@map
└── src/
    ├── client/index.ts      # createPrismaClient(url) + default `db`, Neon adapter
    └── generated/           # `prisma generate` output — gitignored
```

- `package.json` name: `@interviews-tool/infra-prisma-db`, `type: module`.
- Exports `./client` (and `./package.json`), matching the `infra-db` export shape.
- Dependencies: `@prisma/client`, `@prisma/adapter-neon`; devDep: `prisma`.
- Dev deps for workspace wiring: `@interviews-tool/config`, `typescript`, `tsdown` (build)
  following the `infra-db` conventions.

### `prisma/schema.prisma`

- `datasource db` → `provider = "postgresql"`, `url = env("DATABASE_URL")` (same Neon URL).
- `generator client` → `provider = "prisma-client"`, `output = "../src/generated"`, driver
  adapters enabled.
- Models come from `prisma db pull` (introspection). Because tables are prefixed and
  snake*case, introspection yields models like `interviews_tool_hiring_process`; tidy the
  key models with clean names + `@@map("interviews_tool*...")`and`@map` on columns so the
  client reads nicely while pointing at the real tables. Relations are preserved as
  introspected.

### `src/client/index.ts`

Mirrors `infra-db/src/client/index.ts`:

```ts
import { PrismaNeon } from "@prisma/adapter-neon";
import { PrismaClient } from "../generated/client";

export const createPrismaClient = (databaseUrl: string) => {
  const adapter = new PrismaNeon({ connectionString: databaseUrl });
  return new PrismaClient({ adapter });
};

export const db = createPrismaClient(process.env.DATABASE_URL || "");
```

## Central workflow (the requested flow)

1. `prisma db pull` — introspect current Neon, write models into `schema.prisma`.
2. `prisma generate` — generate the typed Prisma Client (autocomplete).
3. Consume `createPrismaClient` / `db` from `src/client`.

Scripts (run through dotenvx, like the existing `db:*`):

- Package: `prisma:pull` → `prisma db pull`, `prisma:generate` → `prisma generate`,
  `prisma:studio` → `prisma studio` (server/Node only).
- Root convenience: `prisma:pull` / `prisma:generate` filtered to `-F @interviews-tool/infra-prisma-db`,
  wrapped in `dotenvx run -f apps/server/.env -- ...`.

## Migrations (out of build scope, documented only)

To honor "change nothing," no migrations run against the DB in this iteration. The package
README documents the **safe baseline workflow** for future Prisma-authored changes:
`prisma db pull` → `prisma migrate diff --from-empty --to-schema-datamodel` → capture as the
initial migration → `prisma migrate resolve --applied` to baseline without executing. Nothing
is run against production tables. This satisfies the "generate tables/migrations" interest
without risk, as a documented follow-up.

## Demo scope (approved: server first, edge later)

- **Iteration 1 (this spec):** package + typed client + a **new, additive Elysia route** in
  `apps/server` that reads the same hiring processes via Prisma (parallel to the existing
  Drizzle route). Bun runtime → near-zero risk. Proves Drizzle-or-Prisma over the same tables.
- **Iteration 2 (documented next step):** a `apps/web` server function (edge/Cloudflare
  Workers) reading via Prisma, to prove edge parity. Deferred because bundling Prisma's WASM
  with Vite/Workers may need iteration; documented, not blocking.

## Verification

- `prisma db pull` succeeds against Neon and produces models for the existing tables.
- `prisma generate` produces a client; TypeScript autocomplete works on model fields.
- The new Elysia route returns the same hiring-process data as the Drizzle route (compare
  output side by side) for the same user.
- `bun run check-types` passes across the workspace; no existing package or consumer changed.

## Risks

- **Edge/WASM bundling** (deferred to iteration 2) — main technical risk; isolated from
  iteration 1.
- **Introspected model naming** — prefixed/snake_case tables produce ugly default names;
  mitigated with `@@map`/`@map` on the key models.
- **Version/flags drift** — exact driver-adapter flags depend on the installed Prisma
  version; pinned at implementation time.

## Out of scope

- Changing or migrating existing tables.
- Replacing Drizzle anywhere.
- Repositories/mappers/use-case layer for Prisma (MVP: client + one demo route only).
- Edge (Cloudflare Workers) Prisma demo — deferred to iteration 2.
